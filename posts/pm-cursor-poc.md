# 产品经理如何用 Cursor 做 POC

> 从构思到可交互 Demo，一个"极客 PM"的工程实践分享。

## 为什么 PM 要自己做 POC？

"你画个原型，开发来评估"——这是传统的产品开发流程。但在 AI 产品领域，这个流程有一个致命问题：

**原型无法表达 AI 能力的动态性。**

一个静态的 Figma 原型，无法让团队感受到：
- AI 实时分析流水的流畅度
- 多轮对话的交互体验
- 风险预警的告警效果

所以我选择：**自己做可交互的 POC Demo。**

## Cursor + Vibe Coding

Cursor 是我目前的主力开发工具。作为产品经理，我的"编程"方式和传统开发者不同：

### 1. 自然语言驱动开发

我不写代码，我写需求：

```
帮我创建一个企业流水分析的仪表盘，包含：
- 左侧：交易流水时间线
- 右侧：风险雷达图
- 底部：异常交易高亮表格
- 整体暗色主题，参考 Linear.app 的设计风格
```

Cursor 会直接生成完整的前端代码。

### 2. 迭代式优化

第一版通常不完美，但足以展示想法。然后持续优化：

```
这个雷达图的动画太生硬了，改成：
1. 页面加载后延迟0.5秒开始绘制
2. 每个维度从0到目标值有一个弹性过渡
3. 绘制过程中带一个淡蓝色的发光轨迹效果
```

### 3. 快速验证交互假设

Cursor 让我能在几分钟内验证一个交互假设：

**假设：** 用户希望通过拖拽选择时间范围来筛选交易

```
添加一个时间范围选择器：
- 类似股票K线图的区间选择
- 拖拽两端可以调整范围
- 选中范围内的交易高亮显示
- 范围外的交易变灰
```

10分钟后，我就能拿着可交互的 Demo 去跟团队讨论这个方案是否可行。

## 半天做出一个完整 Demo

以"智能营销平台"的客户匹配功能为例，我用半天时间做出了：

| 时间 | 产出 |
|------|------|
| 9:00-10:00 | 梳理核心功能点，写出自然语言需求 |
| 10:00-11:30 | Cursor 生成基础页面框架 |
| 11:30-12:30 | 优化交互细节和视觉效果 |
| 14:00-15:00 | 接入模拟数据，跑通完整流程 |
| 15:00-15:30 | 部署到内网，准备演示 |

下午的评审会上，团队直接在 Demo 上讨论需求，效率比看原型图提升了 3 倍。

## PM 做 POC 的边界

**适合做的：**
- UI/UX 交互验证
- 数据展示方案探索
- AI 能力效果演示
- 内部工具快速搭建

**不适合做的：**
- 生产级系统（那是开发的活）
- 性能敏感的功能
- 涉及复杂后端逻辑的场景

## 写在最后

我不追求写出"好代码"，我追求**快速传达产品想法**。Cursor 让产品经理有了"动手验证"的能力，这不是替代开发，而是**缩短从想法到验证的距离**。

这个网站本身，就是我自己用类似方式搭建的——一个产品经理的"全栈实践"。
